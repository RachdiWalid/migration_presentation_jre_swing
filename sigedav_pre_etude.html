01 — Contexte
Contexte et enjeux de modernisation de SIGEDAV
SIGEDAV est l'application centrale de gestion des droits d'auteur de l'ADAGP. Conçue sur des technologies des années 2000, elle accumule une dette technique significative qui fragilise à la fois la sécurité, la maintenabilité et la productivité des équipes.

Stack technologique existant
🔴 Composants techniques actuels
  • Langage : Java 8 — EOL depuis janvier 2019, aucun correctif de sécurité Oracle sans licence payante
  • Interface : Swing — bibliothèque graphique de 1997, rendu géré par la JVM, look & feel figé
  • Back-end : EJB 2/3 — spécification J2EE lourde, nécessite un serveur JBoss/WebLogic coûteux
  • Base de données : Oracle — licence propriétaire onéreuse, forte dépendance éditeur
  • Fichiers complémentaires : Access .mdb/.accdb — données non structurées, risque de perte élevé
  • Déploiement : JAR/EAR monolithique — arrêt global obligatoire à chaque mise à jour
🟢 Périmètre fonctionnel concerné
  • Gestion du catalogue d'œuvres protégées et de leurs métadonnées
  • Référentiel des créateurs, ayants droit et mandats
  • Gestion des licences d'utilisation et de leurs périmètres
  • Calcul et suivi des reversements de droits
  • Reporting interne et tableaux de bord de pilotage
  • Gestion documentaire associée aux dossiers

🏭

📈

03 — Flutter
Flutter : technologie front-end pour SIGEDAV
Flutter est le framework UI open source de Google permettant de créer des applications compilées nativement pour Windows, macOS, Linux, Web, iOS et Android depuis un unique codebase Dart. Il représente une rupture radicale avec Swing dans la qualité d'expérience utilisateur.
Présentation technique de Flutter
  • Créé par Google en 2018 — version 3.x stable et mature depuis 2022, plus de 1 500 contributeurs actifs sur GitHub
  • Langage Dart — fortement typé, null-safety, syntaxe proche de Java/TypeScript — courbe d'apprentissage courte pour les développeurs Java
  • Moteur graphique propre — Impeller (successeur de Skia) — rendu 60/120 fps indépendant du rendu natif OS, pixel perfect sur toutes les plateformes
  • Compilation AOT (Ahead-Of-Time) — l'application est compilée en binaire natif, pas d'interpréteur au runtime — performances proches du natif
  • Architecture widget tree — tout est widget, composable et testable — séparation claire UI / logique
  • Material 3 + Cupertino — composants natifs prêts à l'emploi, thème centralisé, dark mode intégré
  • Hot reload — rechargement de l'UI en millisecondes sans perdre l'état — productivité de développement x2 vs Swing
  • Écosystème pub.dev — plus de 40 000 packages disponibles — D&D, PDF, impression, AD/LDAP, file picker…
Pourquoi Flutter est le bon choix pour SIGEDAV
  • Interfaces riches desktop : formulaires complexes multi-champs, tableaux triables et paginés, D&D natif, sélection multiple — tous les besoins ADAGP couverts nativement
  • Expérience desktop Windows : compilation Win32 native — exécutable autonome sans JVM, sans navigateur, sans dépendance runtime à déployer
  • Pérennité technologique : soutenu par Google avec un budget R&D conséquent, roadmap publique transparente, Flutter est utilisé en production par Google, BMW, eBay, Alibaba
  • Migration progressive : Flutter communique exclusivement via HTTP — il peut se brancher sur une API Spring Boot coexistant avec l'ancien système EJB pendant la transition
  • Un codebase = 5 plateformes : si l'ADAGP souhaite ouvrir l'accès web ou mobile à l'avenir, le même code Flutter compile sans modification majeure
  • Testabilité intégrée : flutter_test au SDK — tests unitaires Dart + widget tests — pipeline CI/CD complet sans dépendance externe

04 — Analyse comparative
Analyse comparative approfondie des technologies front-end
Cette section compare de manière exhaustive les trois options technologiques envisageables pour le front-end de SIGEDAV : Flutter, JavaFX et une SPA Web (Angular/React). L'analyse couvre 13 critères déterminants pour le succès du projet ADAGP.
Analyse forces / vigilances Flutter
✅ Forces — arguments en faveur de Flutter
  • Productivité ×2 vs Swing : hot reload (rechargement en <500ms), composants Material 3 prêts à l'emploi, scaffolding rapide — réduit significativement le temps de développement UI
  • Rendu pixel-perfect 60/120fps : le moteur graphique Impeller de Flutter bypasse les APIs de rendu OS — performances constantes sur tous les Windows, pas de lags liés à la JVM
  • Zéro runtime à déployer : l'exécutable Flutter Windows est un binaire autonome — plus de JVM 8 à maintenir, à mettre à jour, à licencier sur chaque poste
  • Testabilité native : flutter_test SDK intégré permet des tests unitaires Dart ET des widget tests simulant des interactions UI — couverture 100% atteignable sans framework externe
  • Évolutivité multiplateforme : si l'ADAGP souhaite exposer SIGEDAV en web ou mobile, le même code Flutter compile sans réécriture — investissement pérenne
  • Adoption industrielle massive : utilisé en production par Google Pay, BMW, eBay, Alibaba, Nubank (50M+ utilisateurs) — maturité prouvée à grande échelle
  • Communauté et recrutement : 550 000+ développeurs Flutter dans le monde (enquête Stack Overflow 2023), bassin de recrutement large et en croissance rapide
  • Intégration AD/LDAP : via l'API Spring Boot — Flutter n'accède pas directement au SI, tout passe par l'API sécurisée — modèle de sécurité propre et centralisé
⚠️ Points de vigilance — à anticiper dès le démarrage
  • Courbe d'apprentissage Dart : estimée à 2-3 jours pour un développeur Java senior — la syntaxe est proche (orientée objet, typé fort), mais le paradigme "tout est widget" nécessite une acculturation. Mitigation : formation dédiée en phase 3 + PoC guidé
  • Flutter Desktop — maturité moindre que mobile : stable depuis mars 2022, l'écosystème desktop est plus jeune que mobile. Certains packages n'ont pas encore de support Windows. Mitigation : audit des packages nécessaires avant démarrage
  • Intégrations OS spécifiques : impression native Windows, accès périphériques spéciaux, intégration COM — nécessitent des "platform channels" Dart↔Win32. Mitigation : cartographie des besoins en phase 1
  • API REST prérequise : Flutter communique exclusivement en HTTP. Une API Spring Boot bien conçue est un prérequis absolu avant de démarrer le front. Mitigation : démarrer le back-end en phase 2 avant le front en phase 3

Aucun risque de disparition à l'horizon 10 ans.
Migration depuis Swing ⭐⭐⭐⭐⭐ — Rupture propre via l'API REST : Flutter remplace Swing composant par composant sans toucher au back-end EJB existant. Coexistence possible pendant la transition. Le modèle mental "widget" est différent mais cohérent. ⭐⭐⭐ — Même écosystème JVM mais migration à refaire entièrement malgré tout. Risque de perpétuer des patterns Swing dans JavaFX. Pas d'avantage décisif sur Flutter pour la migration. ⭐⭐ — Rupture technologique totale : Java → TypeScript/JavaScript, Swing → HTML/CSS, paradigmes complètement différents. Courbe d'apprentissage la plus longue pour une équipe Java.
Conformité RGPD & sécurité ⭐⭐⭐⭐⭐ — Flutter ne stocke pas de données sensibles côté client. Tout passe par l'API Spring Boot sécurisée. Tokens JWT en mémoire (pas de localStorage). Modèle de sécurité propre et auditables. ⭐⭐⭐ — Même niveau de sécurité atteignable mais nécessite davantage de configuration manuelle (pas de framework de sécurité intégré côté JavaFX). ⭐⭐⭐ — Le navigateur introduit des surfaces d'attaque supplémentaires (XSS, CSRF, localStorage). Nécessite une politique CSP stricte et une gestion fine des cookies. Conformité plus complexe à assurer.
Accès fonctions OS desktop ⭐⭐⭐⭐ — Platform channels Dart↔C++ pour les intégrations OS spécifiques. Impression, clipboard, file system, registres Windows — accessibles via plugins ou développement natif. Quelques packages non encore matures sur Windows. ⭐⭐⭐⭐⭐ — Accès direct aux APIs Java/JVM et donc à l'ensemble des APIs Windows via JNI. Intégration la plus facile pour les cas d'impression native Windows ou accès COM. ⭐⭐ — Accès OS limité aux APIs navigateur (Web API). Impression via CSS. Pas d'accès natif au système de fichiers OS (sauf File System Access API, support partiel). Electron nécessaire pour aller plus loin.
Coût total de possession (TCO) ✅ Faible — Open source MIT, 1 codebase pour 5 plateformes, pas de runtime à licencier, maintenance simplifiée par le typage fort et les tests intégrés. ⚠️ Moyen — OpenJFX gratuit mais compétences rares (coût RH élevé), JVM à maintenir sur chaque poste, écosystème commercial (SceneBuilder). ✅ Faible — Open source, large communauté. Mais nécessite un serveur web, une infrastructure différente du desktop, et des compétences différentes de l'équipe Java actuelle.
Intégration back-end Spring Boot ⭐⭐⭐⭐⭐ — Couplage via API REST uniquement. Contrat OpenAPI 3 partagé. Génération de client Dart depuis le YAML OpenAPI (openapi-generator). Dio comme client HTTP avec intercepteurs JWT. ⭐⭐⭐⭐ — Intégration Java native possible (même JVM). Mais couplage plus fort risqué. Appels REST via HttpClient Java ou RestTemplate. ⭐⭐⭐⭐⭐ — Intégration REST native, génération TypeScript depuis OpenAPI (ng-openapi-gen, openapi-typescript). Fetch/Axios bien maîtrisés.
Verdict Smartpoint : Flutter est le candidat recommandé pour SIGEDAV. Il offre le meilleur équilibre entre performance desktop, richesse des composants, pérennité technologique et coût total de possession. JavaFX présente un risque RH majeur (compétences rares) sans apporter d'avantage décisif sur Flutter. La SPA Web reste une alternative viable si l'accès navigateur devient prioritaire, mais implique une rupture technologique plus profonde pour l'équipe actuelle et des limitations sur les interactions desktop natives.

08 — Conclusion
Conclusion et recommandations préliminaires
Cette pré-étude établit un cadrage architectural fondé sur une analyse rigoureuse des irritants et une évaluation comparative approfondie des technologies candidates. Les recommandations ci-dessous sont préliminaires et seront affinées lors de la mission d'audit.
Synthèse des constats
  • La dette technique est critique et urgente : Java 8 EoL + EJB + Oracle constituent un risque de sécurité, de performance et de coût croissant — l'inaction n'est plus une option viable
  • Flutter est le meilleur choix front-end : performance desktop native, richesse des composants, pérennité Google, TCO faible — l'analyse comparative sur 13 critères le confirme face à JavaFX et aux SPA Web
  • Spring Boot 3 + PostgreSQL sont le standard du marché : stack éprouvée, open source, sans licence éditeur, avec un bassin de compétences très large en France
  • La migration progressive est la seule trajectoire acceptable : un big bang sur un système de production critique est un risque inacceptable — chaque phase doit livrer de la valeur indépendamment
  • Les fichiers Access sont la priorité numéro 1 : ils constituent le risque de perte de données métier le plus élevé et doivent être traités en premier
  • La conformité RGPD sera atteinte by design : l'architecture cible (RLS PostgreSQL, JWT, audit log Spring) intègre la conformité à chaque couche sans surcouche ultérieure
Prochaines étapes recommandées

Message final aux parties prenantes ADAGP
Ce document constitue une pré-étude et non un document d'architecture définitif. Les choix technologiques présentés sont des recommandations préliminaires fondées sur les informations disponibles à ce stade et sur l'expertise Smartpoint en modernisation de SI.

Ce que confirme cette pré-étude

  • La dette technique de SIGEDAV rend la modernisation incontournable à court terme
  • Flutter est une technologie mature et pertinente pour remplacer Swing sur Windows

  • Une migration progressive par domaines est la seule trajectoire acceptable
